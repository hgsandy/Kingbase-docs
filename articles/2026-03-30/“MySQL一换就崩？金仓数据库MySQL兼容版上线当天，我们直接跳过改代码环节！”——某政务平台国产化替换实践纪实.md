### 一、项目背景：不是“能不能换”，而是“敢不敢今晚就切”

这个系统具有典型政企级应用特征：  
- 所有前端调用均基于MyBatis框架，Mapper XML中嵌入大量MySQL特有语法：`ON DUPLICATE KEY UPDATE`、`LIMIT`子句置于`DELETE`语句末尾、`ENUM('draft','submit','approve')`类型字段广泛使用；  
- 后台定时任务依赖MySQL Event Scheduler实现历史数据自动归档；  
- 报表模块采用`GROUP BY WITH ROLLUP`生成五级嵌套汇总结果，前端BI工具完全依赖该结构进行可视化渲染。

此前听厂商介绍“高度兼容”，我只当是常规技术话术。直到查阅金仓发布的《MySQL兼容能力白皮书》，看到其中明确列出支持项：“**支持INSERT ON DUPLICATE KEY、REPLACE、LOAD DATA INFILE、用户变量@var、Event Scheduler事件调度器**”，我才真正将注意力从“重写DAO层”移开。

但真正带来挑战的，是客户临时追加的一条关键要求：**所有含身份证号、手机号等敏感信息的字段，必须在数据库层面实现动态脱敏，并确保不影响现有SQL逻辑执行效率与查询响应时间。**  
这意味着：不能通过应用层拦截过滤，也不能由DBA手动创建视图或中间表；必须是一种可配置、可开关、对业务无侵入的原生能力。

### 二、实施过程：开发、金仓工程师与DBA三方协同推进

#### ▶ 方案研讨：首次会议未见PPT，只见实时适配分析脚本  
我整理出Mapper中最典型的17条强MySQL风格SQL，例如这条融合`INTERVAL`与用户变量的复杂更新语句：  
```sql
SET @last_month = DATE_SUB(CURDATE(), INTERVAL 1 MONTH);  
UPDATE biz_order SET status='archived'   
WHERE create_time < @last_month AND status='completed';
```  
金仓工程师老陈没有查阅文档，而是直接开启共享屏幕，输入命令：  
```bash
./kdms_v4 --source=mysql57 --target=kes_v9r3c11 --mode=mysql --analyze ./mapper/
```  
3秒后，系统返回分析报告：**“兼容度99.2%，仅1处需人工确认（用户变量作用域差异）”**。  
他随即展开详情页解释：“KES原生支持`@var`变量机制，但默认启用‘严格变量模式’；只需在连接初始化阶段执行`SET sql_mode='MYSQL40'`即可保持行为一致——无需修改任何Java代码。”  
那一刻，我对“应用不改、平滑迁移”的可行性有了切实信心。

#### ▶ 适配测试：凌晨两点远程协助，精准定位报表异常根源  
测试进入报表模块时，`GROUP BY WITH ROLLUP`返回结果缺失两行汇总数据。我反复检查执行计划、索引状态，甚至准备重写SQL逻辑。  
金仓DBA小王于凌晨一点接入远程会话，快速浏览查询语句后问道：“你执行的是`SELECT a,b,SUM(c) FROM t GROUP BY a,b WITH ROLLUP`？”  
我点头确认。  
他立即执行：  
```sql
SHOW VARIABLES LIKE 'group_concat_max_len';  
-- 发现当前值为1024，而ROLLUP聚合生成的字符串超出限制被截断  
SET SESSION group_concat_max_len = 1048576;  
```  
再次运行，汇总数据完整呈现。  
他边关闭终端边说明：“这是MySQL兼容版本的默认参数设置，在标准MySQL环境中通常已被调高；我们已在部署检查清单中同步纳入此项配置建议。”  
**原来并非功能缺陷，而是因高度贴近原生MySQL带来的细节适配需求。**

#### ▶ 数据脱敏：零代码改造，敏感字段按权限自动掩码  
针对动态脱敏需求，我们原计划在应用层添加`@Sensitive`注解，再由中间件统一处理。但金仓提供的方案更为简洁高效：  
- 在管理控制台启用“动态脱敏策略”，选择`person_info`表中的`id_card`列，设定脱敏规则为“前4位+后4位保留”；  
- 执行授权语句：  
```sql
GRANT SELECT ON person_info TO app_user WITH MASKING;
```  
此后，`app_user`账号查询该字段时，始终返回形如`1101**********1234`的掩码结果；  
管理员账户仍可见原始明文；  
MyBatis的`<resultMap>`映射无需变更，因脱敏发生在**数据封装为网络包之前**，对JDBC驱动完全透明。  
上线前压测结果显示：启用脱敏策略后的平均查询耗时增幅低于0.8%，满足性能红线。客户当场拍板：“按此方案推进，下周正式割接。”
