#### ▶ 适配测试：性能瓶颈源于日期精度设置差异，一次配置即解决

上线前三天，压力测试再次出现异常：单节点QPS达到1200时，人脸特征匹配耗时由120ms骤升至2.3秒。监控显示数据库CPU利用率未达阈值，但慢查询日志集中于如下语句：  
```sql
SELECT id FROM face_record 
WHERE capture_time >= TRUNC(SYSDATE) - 1 
  AND similarity_score > 0.92;
```

团队花费两小时排查执行计划，怀疑索引失效。金仓工程师远程接入后，查看 `capture_time` 字段定义，询问：“Oracle环境中该字段为 `DATE` 类型，但实际存入的是带毫秒精度的 `TIMESTAMP` 数据？”  
我们确认属实。  
他执行命令：  
```sql
SHOW kingbase_date_trunc_precision;
```
返回结果为 `'second'`（默认按秒级截断）。而Oracle中 `TRUNC(SYSDATE)` 默认保留毫秒精度，导致索引范围扫描失效，触发全表扫描。

解决方案简洁明确：  
```sql
SET kingbase_date_trunc_precision = 'millisecond';
```
再次压测，平均响应时间回落至135ms以内。  
他解释道：“这不是缺陷，而是设计取向不同。Oracle默认更激进地保留精度，金仓则优先保障数据一致性与可预测性——但关键控制权，始终交由使用者掌握。”

#### ▶ 问题解决：上线前夜精准优化索引策略，提升系统吞吐能力

上线前最后一晚，系统压力峰值达3500 QPS，连接池满载，`sys_stat_activity` 中出现大量 idle in transaction 状态连接。  
运维反馈：“疑似事务未正常关闭。”  
我们核查代码逻辑，确认事务控制流程无误。金仓团队迅速派出资深架构师，携带性能热力图分析工具直连镜像环境。  
经过10分钟观察，他指出 `face_match_log` 表存在潜在瓶颈：“该表日增记录约500万条，当前仅在 `create_time` 字段建立单列索引，但高频查询条件为 `WHERE status='matched' AND create_time > ?`。由于 `status` 字段基数较低，当前版本优化器对索引选择率判断趋于保守，主动降级为顺序扫描。”

建议措施如下：  
1. 创建复合索引：`CREATE INDEX idx_status_ct ON face_match_log(status, create_time);`  
2. 在过渡阶段添加执行提示：`/*+ IndexScan(face_match_log idx_status_ct) */`  

操作完成后重启服务，再进行压测：连接数稳定维持在180左右，P99响应时间控制在180ms以内。
