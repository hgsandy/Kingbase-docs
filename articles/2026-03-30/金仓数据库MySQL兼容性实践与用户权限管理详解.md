## 一、MySQL兼容性：不止于语法“像”，更在于行为“同”

金仓数据库的MySQL兼容能力并非简单的关键字映射或语法识别，而是构建在**协议级适配、语义级对齐与行为级校准**三层协同架构之上。其设计目标是让存量应用代码实现“零修改”或“极少量调整”即可稳定运行，重点解决三类高频迁移痛点：

### 1. 连接与协议兼容：驱动即插即用  
应用无需更换JDBC/ODBC驱动，亦不需修改连接配置逻辑。经实测验证：  
- MySQL Connector/J 8.0.x 可直连金仓数据库（默认端口5432，协议握手阶段自动识别MySQL兼容模式）；  
- 连接字符串格式保持完全一致：`jdbc:mysql://host:port/dbname?useSSL=false&serverTimezone=Asia/Shanghai`；  
- 支持`mysql-c`等原生C语言客户端库，无需额外编译适配中间层。  
> ✅ 实践佐证：某大型车企车联网安全认证平台，在未调整Spring Boot数据源配置的前提下，仅将数据库地址指向金仓实例，`spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver`保持不变，服务启动成功且功能完整。

### 2. SQL语法与对象定义：表/索引/视图“平滑迁移”  
在MySQL兼容模式下，金仓数据库支持以下关键语法特性：  
- **DDL语句**：`CREATE TABLE t(id INT PRIMARY KEY, name VARCHAR(50) NOT NULL) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4` 中的`ENGINE`与`CHARSET`参数被自动识别，并映射为对应存储策略与字符集配置；  
- **DML语句**：`INSERT IGNORE INTO`、`REPLACE INTO`、`ON DUPLICATE KEY UPDATE` 等常见冲突处理语法均可正常执行；  
- **函数与表达式**：`NOW()`、`DATE_FORMAT()`、`CONCAT()`、`IFNULL()` 等200余种常用函数，在返回值类型、空值处理、时区行为等方面与MySQL 5.7高度一致；  
- **注释风格**：全面支持`-- 注释`、`# 注释`、`/* 多行注释 */`三种MySQL标准注释格式解析。

> ⚠️ 兼容提示：`AUTO_INCREMENT`列在底层由序列机制实现，建表时自动创建关联序列对象，但`SHOW CREATE TABLE`输出仍保留`AUTO_INCREMENT`关键词，确保开发人员感知无差异。

### 3. 行为级一致性：事务、锁机制与错误反馈精准对齐  
这是衡量兼容深度的关键维度。金仓数据库通过内核级行为模拟，保障如下核心能力：  
- **事务隔离控制**：在`REPEATABLE READ`隔离级别下，`SELECT ... FOR UPDATE` 对当前读取行施加行级写锁，且不阻塞其他会话对非冲突数据的普通读取（如`SELECT * FROM t WHERE id = 2`），该行为与MySQL InnoDB引擎保持一致；  
- **错误码映射机制**：当执行`INSERT INTO t VALUES(1)`而主键已存在时，返回标准错误信息 `ERROR 1062 (23000): Duplicate entry '1' for key 'PRIMARY'`，而非泛化SQLSTATE错误码，有效避免应用层异常捕获逻辑失效；  
- **元数据查询兼容性**：支持`SELECT * FROM information_schema.tables`、`SHOW TABLES`、`SHOW COLUMNS FROM t`等常用元数据查询语句，字段结构与返回结果格式贴近MySQL习惯，降低适配成本。

## 二、用户权限管理：遵循最小权限原则的全流程实践

权限管理是数据库安全治理的核心环节。金仓数据库提供基于角色的细粒度权限模型，支持从数据库级、模式级、表级直至列级与行级的逐层授权控制，满足金融、政务、能源等高敏感行业对访问可控性的严苛要求。

### 1. 用户与角色体系设计  
系统支持创建独立登录用户，并可通过角色进行权限聚合管理。例如：  
```sql
CREATE USER app_user WITH PASSWORD 'StrongPass@2024';
CREATE ROLE read_only_role;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO read_only_role;
GRANT read_only_role TO app_user;
```
上述操作赋予用户只读权限，且后续新增表自动继承该权限，显著提升运维效率。

### 2. 权限分配与动态调整  
除静态授权外，还支持基于会话上下文的动态权限控制。例如结合应用标签（application_name）或客户端IP段实施条件化访问限制，配合`kingbase.conf`中的访问控制规则，形成多层防护体系。

### 3. 操作审计与权限回收  
启用系统级审计功能后，所有`GRANT`、`REVOKE`、`CREATE USER`等权限变更操作均被记录至审计日志，支持按时间范围、操作类型、执行用户等多维检索。当人员岗位变动或项目终止时，可一键回收全部相关权限，杜绝权限残留风险。

### 4. 默认权限与安全基线建议  
新创建用户默认不具备任何对象操作权限，须显式授权方可访问。推荐采用如下安全基线配置：  
- 禁用匿名用户与空密码账户；  
- 启用密码复杂度策略（长度≥8位，含大小写字母、数字及特殊字符）；  
- 定期轮换高权限账户密码，周期不超过90天；  
- 生产环境关闭`superuser`权限直连，改由专用运维通道审批执行。
