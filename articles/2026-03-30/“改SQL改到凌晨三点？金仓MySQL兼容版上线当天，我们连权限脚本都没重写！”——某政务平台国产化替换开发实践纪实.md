### 一、项目背景：不是“能不能换”，而是“敢不敢动生产权限”

我们承接的是某省级政务“一网通办”平台的底层数据中台建设任务，承载企业注册、证照核发、信用查询等关键政务服务，日均接口调用量超120万次。系统长期依托MySQL构建，开发与运维已形成高度适配的工作范式：

- 权限控制精细至字段级别，例如“仅允许SELECT t_user.name, t_user.mobile WHERE dept_id = ?”；
- 大量采用`CREATE USER 'app_rw'@'%' IDENTIFIED BY 'xxx'` + `GRANT SELECT, INSERT ON db.* TO 'app_rw'@'%'`组合实现访问控制；
- 还有十余个自动化运维脚本，依赖`mysql -e "SHOW GRANTS FOR 'monitor'@'localhost'"`完成权限巡检与变更审计。

当国产化替换消息传来，我连续两晚未能入眠。担忧并非来自SQL语法适配难度，而是源于权限配置一旦出现偏差——  
✅ 表结构可完整导出；  
✅ 数据同步过程可控；  
❌ 但任意一行权限语句错误，都可能引发报表异常、流程中断，甚至导致厅局级政务服务系统响应迟滞或部分功能不可用。

更关键的是：**生产环境的权限体系无法试错**。政务系统需满足等保三级及以上要求，“谁授权、何时授、授了什么”每一条操作均须完整留痕并纳入审计日志。而不同数据库在权限模型设计上存在显著差异——如嵌套授权机制、角色继承逻辑、列级权限支持方式等。我们无法预判金仓在权限校验层面是否与MySQL保持行为一致。

当时我内心的真实想法是：**“宁可重写千行业务逻辑，也不愿手动调试一行权限语句。”**

### 二、实施过程：从“技术疑虑”到“协作增效”

#### ▶ 方案研讨：首次联调即见成效，兼容性工具提升评估效率

我们组织了一次四方协同会议（我方开发+DBA、金仓售前工程师、金仓研发工程师），将全部MySQL权限初始化脚本、历史GRANT执行日志、权限层级关系图一并提供，并梳理出17项待确认的技术细节，例如：

> “`CREATE USER 'api'@'10.%.%.%' IDENTIFIED BY 'pwd' PASSWORD EXPIRE NEVER` 中的`PASSWORD EXPIRE`子句，金仓是否支持？”  
> “`GRANT EXECUTE ON PROCEDURE db.proc_check_credit TO 'svc'@'%'` 类型的存储过程执行授权，在金仓中应映射为FUNCTION还是ROUTINE？”

金仓研发工程师未作理论解释，而是直接打开KStudio客户端，启用【MySQL兼容模式】，并将我们的`init_privileges.sql`脚本拖入校验窗口——  
**3秒后，界面弹出绿色提示：“语法校验通过，兼容性匹配度98.7%”。**

他随即指出：“`PASSWORD EXPIRE NEVER`属于MySQL 8.0+特性，金仓虽不主动解析该参数，但不影响用户创建；`EXECUTE ON PROCEDURE`统一按标准SQL规范映射为`EXECUTE ON FUNCTION`，无需修改原有脚本。”  
我追问：“那类似`GRANT SELECT (id,name) ON t_user TO 'report'@'%'`这样的列级权限呢？”  
他回应：“完全支持。此外，金仓还支持通过角色间接授予列级权限，例如先给角色A赋权，再将角色A赋予用户B，整条授权链路可在审计日志中清晰追溯。”

那一刻，我截下屏幕画面，发至内部开发群：“各位，今晚可以按时下班了。”

#### ▶ 适配测试：定位隐性差异，精准解决大小写敏感问题

上线前夜压测期间，监控平台触发告警：**3个高频服务接口返回500错误，日志中频繁出现`ERROR: permission denied for table xxx`。**  
团队排查半小时未果——用户具备对应表的SELECT权限，表对象存在，连接配置无误。DBA通过网络抓包发现，应用发送的SQL中含有数据库名前缀`FROM db_name.t_user`，而金仓默认未开启`search_path`自动绑定库名。

进一步分析发现，金仓在权限校验过程中对数据库名称实行严格大小写匹配。我们在MySQL环境中使用的库名为`db_name`，但在金仓部署时因历史命名规范统一采用大写形式`DB_NAME`。  
—— 正是这一处大小写差异，导致权限判定失败。

凌晨一点，金仓技术支持工程师远程接入，共享屏幕后迅速执行两条诊断命令：
```sql
-- 查询当前用户实际生效的角色与权限范围
SELECT * FROM sys_roles WHERE rolname = 'app_ro';

-- 查询目标表所属命名空间（即数据库名）
SELECT nspname FROM sys_namespace JOIN sys_class ON sys_namespace.oid = sys_class.relnamespace 
WHERE relname = 't_user';
```
结果显示：表确属`DB_NAME`命名空间。他随即建议：“请将脚本中所有`db_name.xxx`统一替换为`DB_NAME.xxx`，或在数据库连接串中添加`currentSchema=DB_NAME`参数。”

我问：“就这么简单？”  
他答：“是的。另外提醒，金仓的`sys_hba.conf`支持基于IP段、用户名、数据库名三维度的访问控制策略，权限粒度比传统host-based auth更细致，如需模板我可即时提供。”

当天凌晨三点，我们完成脚本更新，尚未吃完泡面，压测结果已全面恢复正常。

#### ▶ 问题解决：上线前紧急协同，共建标准化权限管理流程

最紧张的时刻出现在上线前两小时——CI/CD流水线中的自动化权限部署脚本执行失败。  
原Python脚本依据配置动态生成GRANT语句，逻辑如下：
```python
grant_sql = f"GRANT {priv} ON {schema}.{table} TO '{user}'@'{host}'"
```
但金仓遵循标准SQL语法，不识别`@'host'`格式，且要求用户必须预先存在，不能在授权语句中同步创建。

金仓架构师现场登录CI服务器，编写适配脚本：
```python
# 金仓专用：分步执行，确保幂等性
cursor.execute(f"CREATE USER IF NOT EXISTS {user} PASSWORD '{pwd}'")
cursor.execute(f"GRANT {priv} ON TABLE {schema}.{table} TO {user}")
```
同时补充判断逻辑：“后续新增权限只需运行该脚本，重复执行不会产生冲突。”

他解释道：“MySQL中`GRANT ... TO 'u'@'h'`本质是建用户+授权合并操作，而金仓拆分为两个独立步骤，有助于规避因用户创建失败导致的权限残留风险，提升整体安全性。”

当晚，团队在办公室点了六份黄焖鸡，边用餐边听取金仓专家分享权限模型设计理念：  
**“并非简单复刻MySQL语法，而是以更严谨的标准SQL为基础，推动权限管理从经验驱动转向策略驱动。”**
