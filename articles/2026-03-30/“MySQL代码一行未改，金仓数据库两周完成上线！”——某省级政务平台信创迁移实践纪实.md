### 二、实施过程：不是“单兵作战”，而是“协同共建式技术交付”

#### ▶ 方案研讨：一页纸问题清单，换来自动化适配工具支持  
首次线上技术对接会上，我们将系统中所有MySQL特有或非标准SQL写法整理成一份Word文档提交给金仓工程师：`IFNULL()`、`CONCAT_WS()`、`UNIX_TIMESTAMP()`、`STR_TO_DATE()`等共计23类典型表达。  
对方并未要求逐条重写，而是共享屏幕演示**KDMS结构迁移工具**——粘贴任意一段MySQL SQL，点击“MySQL→KES适配”，3秒内输出可执行脚本，并附带详细注释说明转换逻辑，例如：“`IFNULL(a,b)` → `COALESCE(a,b)`（兼容多种模式）”。  
尤为实用的是，工具还能识别MyBatis中`${xxx}`拼接方式潜在的SQL注入风险，并高亮提示：“建议改用参数化占位符`#{xxx}`以提升安全性”。  
**这背后体现的是长期积累的迁移经验沉淀，而非临时应对的技术补丁。**

#### ▶ 适配测试：耗时两小时未定位的问题，远程十分钟完成根因分析  
压测阶段出现一个隐蔽偏差：报表模块“近7天办件趋势图”前端展示数据比MySQL环境少1条记录。  
我反复比对SQL执行计划、JDBC连接参数、系统时区设置等十余项配置，均未发现差异。  
金仓工程师小李远程接入后，第一句就问：“应用服务器设置了`Asia/Shanghai`时区，数据库实例当前使用的时区是？”  
经查，KES初始化默认采用UTC时区。他随即执行命令：  
```sql
ALTER DATABASE kingbase SET timezone = 'Asia/Shanghai';
```  
再次运行报表查询，数据完全对齐。  
他补充道：“MySQL默认继承操作系统时区，而KES遵循国际通用默认值设定。此类细节差异已在《MySQL迁移常见问题指南》第3.2节中专项说明。”  
**真正的兼容，不在于机械复刻，而在于降低开发者理解成本，减少跨平台语义翻译负担。**

#### ▶ 问题解决：上线前关键阶段的性能优化实战  
上线前72小时，系统在并发2000请求压力下TPS骤降。监控显示主库CPU使用率达92%，但慢日志中高频出现的却是形如`SELECT COUNT(*) FROM biz_apply WHERE status=1`这类基础查询。  
金仓架构师老陈视频接入后，未急于分析SQL本身，而是直接询问：“这张表的`status`字段是否建立了独立索引？”  
我们一时语塞——在MySQL实践中，对于仅含0/1/2三种取值的整型字段，通常不会单独建索引。  
老陈调取执行计划指出：“当前执行路径为Bitmap Index Scan on idx_status，但由于现有联合索引`(create_time, status)`中`status`非最左前缀，导致无法高效利用，最终退化为全表扫描。”  
他指导我们创建函数索引：  
```sql
CREATE INDEX idx_status_only ON biz_apply ((status::text));
```  
重新压测后，TPS稳定在3200以上，主库CPU回落至65%左右。  
凌晨三点，我们在会议室分食一桶方便面，他发来信息：“明早八点前，我会将本次索引优化建议与配套自动化巡检脚本打包发送。”
