
上周一刚到公司，开发同事就反馈：“测试环境连接金仓数据库后，无法查询到列车实时状态！”——客户下周就要开展验收工作，若此时出现关键功能异常，将直接影响项目交付节奏。我端着保温杯迅速回到工位，第一反应不是立即查看SQL日志，而是打开手机中金仓技术支持的钉钉群，发送了一条语音消息：“老师您好，ATS主控屏上‘列车位置’字段全部为空，Oracle环境下运行正常，切换至金仓后基础SELECT语句返回空结果……是否是我们某条SQL存在兼容性问题？”

这并非首次遇到“查不到数据”的情况，但此次确实令人高度关注。因为这是某市地铁14号线ATS（列车自动监控系统）国产化适配项目——全国首个基于双网段架构部署的轨道信号系统整体国产化实施方案，也是我们团队本年度重点推进的信创落地任务之一。而金仓数据库，正是本次系统升级的核心支撑组件。

### 一、项目背景：不是简单替换数据库，而是对系统运行能力的一次全面验证，留给测试的时间仅剩20天

简言之，ATS系统承担着地铁运营中枢职能：需以毫秒级响应采集全线列车位置、信号机状态、道岔位置、临时限速等关键运行数据，并据此实时计算进路、生成控制指令、联动联锁设备。此前采用Oracle数据库时，测试过程中主要面临三类典型挑战：

- **环境差异带来的不确定性**：生产环境运行于AIX+Oracle RAC平台，测试环境则部署在Linux+单实例配置下，部分压测场景中出现偶发性超时现象，经排查发现实为操作系统层IO调度策略不同所致，并非SQL逻辑本身存在问题；
- **报表导出适配难度较高**：Oracle环境下导出Excel可自动适配中文列宽，但在测试环境中常出现乱码与列宽错位问题，业务方据此质疑系统稳定性与上线可行性；
- **整体适配周期紧张**：要求在三个月内完成全链路国产化改造，其中分配给测试验证的时间仅有20天，还需预留5天用于UAT及客户签字确认。仅兼容性验证一项即涉及287个核心接口、43类报警规则、6套联锁逻辑脚本，以及大量嵌套在存储过程中的状态处理逻辑……

更关键的是，客户明确提出三项基本运行要求：“响应性能不低于原有系统、告警信息不遗漏、主控界面无中断显示”——这些不仅是功能指标，更是保障运营连续性的基础条件。

### 二、测试实施过程：不是单点排查问题，而是依托协同机制实现高效适配

#### ▶ 方案制定阶段：结合历史经验梳理风险点，金仓专家补充多项关键验证项

首次三方会议（我方、开发团队、金仓技术专家）中，我没有按常规介绍测试流程，而是直接投屏展示了整理好的《Oracle ATS测试常见适配问题清单》：
- “WHERE status IN ('RUN','STO') AND train_id LIKE '01%'” 在Oracle中可命中索引，但在金仓中执行计划显示为全表扫描；
- “TO_CHAR(sysdate, 'YYYYMMDDHH24MISS')” 在Oracle中输出格式为‘20250715103022’，金仓默认返回带空格格式；
- “INSERT /*+ APPEND */ INTO alarm_log…” 并行插入操作在Oracle中由系统自动优化分片，在金仓中需显式启用批量插入模式。

金仓专家老张听完后并未简单回应“已知问题”，而是现场记录并提出补充建议：“上述三个场景，金仓KDMS迁移工具已内置相应校验能力；此外，建议增加两项边界验证：一是双网切换过程中的事务一致性保障（ATS系统设定5秒心跳检测机制），二是麒麟操作系统环境下JDBC驱动与金仓SSL握手超时后的重试逻辑。”

那一刻我深刻体会到：此次合作并非单纯的产品引入，而是围绕真实业务场景展开的技术共建。

#### ▶ 适配与兼容测试阶段：“Excel导出异常”问题耗时10小时，金仓工程师远程调整字符集参数，5分钟恢复正常使用

报表模块是适配过程中最棘手的部分。ATS每日需生成《运营日报》《故障分析表》《区间占用热力图》三类Excel文件，原Oracle环境导出稳定可靠，更换为金仓后却出现以下问题：
- 中文列名显示为“???”；
- 数值型列宽度严重压缩，仅约3像素；
- “计划到达时间”字段中所有“14:30:00”均被转换为“1900-01-01 14:30:00”。

我尝试十余种解决方案：修改JDBC连接字符串添加useUnicode=true参数、调整Tomcat server.xml中URIEncoding设置、使用setNString()方法强制编码……均未奏效。凌晨一点，我携带笔记本前往金仓驻场工程师办公位寻求协助。对方查阅日志后指出：“问题不在代码侧，而在数据库客户端字符集配置缺失。ATS系统调用的是国产WPS Office组件，其读取Excel依赖数据库返回的`character_set_client`值。Oracle默认为AL32UTF8，但当前金仓测试镜像未同步配置该参数。”

随后他远程接入系统，执行以下三条命令：
```sql
SET character_set_client = utf8mb4;
SET character_set_results = utf8mb4;
SET collation_connection = utf8mb4_unicode_ci;
```
刷新报表页面后，中文列名、数值列宽、时间格式全部恢复正常。他还分享了一个实用口诀：“WPS识别字符集，永中关注BOM头，金山适配JDBC参数——后续如遇类似问题，可先确认所用办公软件类型。”

#### ▶ 问题定位与修复阶段：上线前72小时压测中出现偶发延迟，金仓团队连夜提供索引优化方案

最后三天，我们在模拟100列车并发进站场景进行压力测试。监控数据显示：98.5%请求响应时间低于200ms，但仍有1.5%请求延迟集中在800ms至2.3s之间，且无明显规律。

通过日志追踪发现，延迟请求集中于如下SQL语句：
```sql
SELECT * FROM train_status WHERE line_id = ? AND update_time > ? ORDER BY update_time DESC LIMIT 1;
```
该语句用于获取指定线路最新列车状态，Oracle中已建立复合索引`(line_id, update_time)`，金仓也创建了同名索引，但执行计划显示未使用索引，触发全表扫描。

金仓DBA老李深夜接入分析，仅询问一句：“update_time字段使用的数据类型是TIMESTAMP还是DATETIME？”  
我回答：“DATETIME，精度为秒。”  
他随即说明：“此为典型类型适配差异。金仓对DATETIME类型的范围查询索引选择率预估存在一定偏差，建议通过hint强制走索引。”  
并提供优化语句：
```sql
SELECT /*+ INDEX(train_status idx_line_time) */ * FROM train_status ... 
```
加入hint后，压测通过率提升至100%，P99延迟稳定在192ms以内。

后续复盘得知，该优化方案源自金仓针对轨道交通高频查询特征构建的索引推荐模型，老李调用的是模型生成的第七号适配预案。
