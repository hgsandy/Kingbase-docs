### 一、为什么是强制访问控制？——应对能源行业安全治理的“三重挑战”

能源行业信息系统长期面临三类典型安全治理挑战：

- **角色权限交叉挑战**：调度员、运维员、审计员、第三方服务商职责边界清晰但交互频繁，传统自主访问控制（DAC）依赖用户主动授权，存在权限配置冗余、回收滞后、越权风险上升等问题；
- **数据敏感性分层挑战**：场站实时遥测数据（L1）、设备健康诊断数据（L2）、电网调度指令（L3）等具有不同敏感属性，需依据业务属性与管理要求实施差异化保护；
- **标准合规适配挑战**：依据公安部《GB/T 20273-2019 信息安全技术 数据库管理系统安全技术要求》，具备高等级安全能力的数据库系统应支持强制访问控制机制，而多数产品尚处于基础访问控制阶段。

在此背景下，国家能源集团明确技术路径：**新集控系统所采用的数据库需具备原生MAC能力，支持基于安全标记（Security Label）的细粒度访问裁决机制，并能与现有Oracle权限体系保持高度兼容，最大限度降低迁移改造成本。**

### 二、金仓方案：Oracle兼容版MAC的“三阶落地”实践

金仓团队并未另起炉灶设计一套全新模型，而是深入理解Oracle Label Security架构逻辑，在KingbaseES Oracle兼容模式下，构建了**标记定义→策略绑定→运行裁决**的三层MAC实施路径，兼顾技术先进性与工程可行性：

#### ▶ 第一阶：安全标记（Label）的Oracle级兼容定义  
KingbaseES支持Oracle Label Security中常用的数据字典视图与PL/SQL包接口（如`SA_USER_ADMIN`, `SA_LABEL_ADMIN`）。项目中，工程师复用原有Oracle脚本创建安全策略，大幅缩短适配周期：

```sql
-- 创建安全策略（语法兼容Oracle）
BEGIN
  SA_SYSDBA.CREATE_POLICY(
    policy_name => 'ENGY_SEC_POLICY',
    column_name => 'SECURITY_LEVEL'
  );
END;

-- 定义敏感标签（L1-L4对应场站数据等级）
BEGIN
  SA_LABEL_ADMIN.CREATE_LABEL(
    policy_name => 'ENGY_SEC_POLICY',
    label_tag   => 100,
    label_value => 'L1::PUBLIC'
  );
  SA_LABEL_ADMIN.CREATE_LABEL(
    policy_name => 'ENGY_SEC_POLICY',
    label_tag   => 400,
    label_value => 'L4::GRID_CONTROL'
  );
END;
```

> ✅ 关键优势：**权限建模逻辑无需重构**，Oracle DBA可快速上手；所有标签策略通过新增`SECURITY_LEVEL`列注入，不改变原有业务表结构。

#### ▶ 第二阶：策略绑定与动态裁决  
在关键表（如`SCADA_DATA`, `DEVICE_CMD_LOG`）启用策略后，KingbaseES内核在每次DML/SELECT执行前自动调用MAC引擎，比对会话标签（Session Label）与数据行标签（Row Label），依据预设规则进行实时访问裁决：

- 调度中心账号（Session Label = `L4::GRID_CONTROL`）可读写全部场站指令；
- 场站运维员（Session Label = `L2::SITE_MAINTAIN`）仅能读取本场站L1/L2数据，不可访问L3及以上指令；
- 第三方监控平台（Session Label = `L1::MONITOR`）仅允许只读聚合指标，禁止访问原始遥测值。

> ✅ 实测效果：在27省区并发接入压力下，MAC策略平均裁决耗时低于0.8ms，**未引入明显性能开销**，满足集控系统亚秒级响应需求。

#### ▶ 第三阶：与高可用、迁移工具链深度融合  
- **主备集群一致性保障**：MAC策略元数据（如`SA_POLICY_TABLES`, `SA_LEVELS`）随主库事务同步至备库，Failover后策略即时生效；
- **sys_实时同步无损传递**：增量数据同步过程中，`SECURITY_LEVEL`标签字段与业务数据同批次传输，确保总部数据中心数据标记完整性；
- **kingbase全量迁移自动映射**：迁移工具识别Oracle `LABEL`列并自动创建KingbaseES兼容的`SECURITY_LEVEL`列及关联策略，**迁移过程无需人工干预标签配置**。
